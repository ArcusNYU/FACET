# 用于可模块化的 train pipe 工程文件结构:
FACET(root)/
├── facet/                       # 模型 (已有)
│   ├── config.py / config.yaml / lora.py / model.py(核心为class FACETWanModel)
├── data/                        # 数据 (已有)
│   ├── config.yaml / datasets.py / sampler.py / transform.py / ref_sampler.py
│   └── openvid/ ...
|    data文件包含了完整的子数据集->联合数据集->sampler的流程 已经被封装为loader 可以在 root/loader.py中找到
├── trainer/                     # ★ : 训练流水线的"积木"
│   ├── __init__.py
│   ├── setup.py                 # accelerate / seed / cudnn / tf32 / device 初始化
│   ├── config.py                # TrainConfig dataclass + merge(FACETConfig, train.yaml)
│   ├── optim.py                 # build_optimizer / build_lr_scheduler (LoRA 参数过滤)
│   ├── loss.py                  # FlowMatch noise/velocity 目标 + MSE 包装
│   ├── ckpt.py                  # save_lora / save_full_state / load_resume / top-K bookkeeping
│   ├── logger.py                # mlflow + console + run_dir + config snapshot
│   ├── valid.py                 # 验证循环, 调 metrics.py, 保存采样视频
│   └── sum.py                   # 启动期打印可训练参数表
├── visual/                      # 可视化 (已有) + 未来 gradio_app.py
│   ├── loader_visual.py / model_visual.py
├── runs/                        # ★ : 每次训练的所有过程性产物
│   └── run_name/
│       ├── config_snapshot.yaml         # facet+train 两份 yaml 全量快照
│       ├── # env.txt                    # torch/cuda/git sha/world size 暂时不需要
│       ├── params.txt                   # 可训练参数表 (sum.py 输出)
│       ├── logs/                        # mlflow / tb 文件
│       ├── samples/steps_xxx/           # 验证期间生成的小样本视频 + GT
│       ├── metrics.jsonl                # 每特定数量global_steps步/每 epoch 指标
│       └── ckpts/                       # epoch_xxx.safetensors (训练态, 含 optimizer state) 或者 steps_xxx.safetensors
├── ckpt/                        # ★ 要求: 仅存"对外可用"的 LoRA 权重
│   ├── run_name_best.safetensors    # 按 top-K 选出 + 始终保留 latest
│   ├── run_name_last.safetensors    # 每次都覆盖 这样可以在将来支持resume训练
│   └── manifest.json                    # 指向对应 runs/ 目录、复现命令、指标
├── weights/WAN2.2/              # 基座权重 (已有)
├── train.py                     # 入口: 仅做编排, 真正逻辑分散在 trainer/
├── launch_train.py              # 解析 train.yaml -> 拼装 accelerate launch 命令
├── train.yaml                   # 训练超参 (会覆盖 facet/config.yaml 的部分字段)
├── test.py / launch_test.py / test.yaml      # eval 同模式
├── metrics.py                   # 视频质量指标 (FVD / PSNR / SSIM / LPIPS / CLIPSim)
├── loader.py / utils.py / __init__.py

ckpt文件夹中用于保存当前训练run最好的checkpoint 以及一个manifest.json文件 将checkpoint和原run name对应起来
test.py将从ckpt文件夹中获取指定的checkpoint 并进行测试
训练中产生的 数据包括但不限于logs, 指标, 样本, 模型checkpoint 都会被保存到runs文件夹中
也就是每次训练run都需要在runs文件夹中创建一个当前run的子文件夹
取名方式是 日期 + sxxx 的方式  s指的是global steps的总数 也就是我不建议在本项目中以epoch为单位进行记录或者测评
因为模型是在联合数据集上进行训练 所以以global steps为单位更加合理
"日期 + sxxx"是固定的生成 然后这个字段后面还支持加入用户传入的cfg.names作为尾缀
也就是output_root子文件夹在runs中的名称是 "日期 + sxxx + cfg.names"
那么 "output_root"指向这个子文件夹 需要自动在runs中创建该文件夹 
其中的 .yaml / .txt / .jsonl 可以等即将要写入的时候检查在不在 不在则马上创建 
然后 "ckpt_path"则指向 ckpts文件夹 同样的 manifest.json 可以等即将要写入的时候检查在不在 不在则马上创建 
"weight_path"则指向 weights/WAN2.2文件夹 也就是基座权重文件夹
"data_path"目前其实暂时还不需要 因为loader.py已经把这个模块封装好了

上传至mlflow的字段:
train: loss & lr & grad_norm & timestep_dist??? 
val: loss & metrics 

===================================================
train.py启动逻辑 & train.yaml
launch_train.py根据train.yaml中的配置 在引用并覆盖部分 facet/config.yaml中的配置的情况下
拼装accelerate launch 命令 并启动训练 所以是一种很常规的启动方式
对于启动方式可在train.yaml中选择 auto/python/accelerate三种shell命令启动方式 
如果选择了 auto 那么就看 gpu 写的是几个 如果是1个就用python启动 如果是多个就用accelerate启动 

train.yaml需要包含的内容:
路径 例如如上提到的 path
计算底层配置 涉及acc的配置 例如 precision / deterministic / benchmark / tf32 / gradient_checkpoint
训练功能配置 例如 CFG训练(暂时设置为false) / CFG训练时的drop_prob / mlflow训练时采用的cloud logging的名称 名称可以与run name一致
            还包括cached: True 意思就是 caption embedding & target video latent已经提前被编码好了
训练步数配置 例如 epoch & max_steps & warmup_steps 数量 grad_accum_steps / log_every / save_every / eval_every 等等
训练超参配置 例如 lr / weight_decay / betas / eps / grad_clip / scheduler / optimizer / max_grad_norm / batch_size 等等
验证配置  例如 在validation步骤中 需要生成在runs/output_root/samples/steps_xxx/ 中的样本个数

train.yaml中只需要写所有关于训练需要的东西 以及 覆盖facet/config.yaml中的部分内容的覆盖项


===================================================
train.py结构框架 (参照于DiffSynth架构)

trainer文件夹中大致涵盖了部分training pipeline中可被模块化的部分
在书写整个training pipeline的时候 当涉及某一个模块 关联trainer中的部分时
则现在对应的 xxx.py写出 class xxx 然后在train.py中调用这一部分 
不管是写哪部分 如果能够参考DiffSynth的架构 那么尽量参考它 避免因自己设置导致的模型训练不收敛
某些公共工具(不是模块) 例如test.py也可能用得到的函数 可以放在 utils.py中 

train.py  伪代码骨架  (实际实现散到 trainer/*.py)
也就是各个trainer中的子.py文件中的模块需要映射成 trainer的子属性模块 通过 trainer.xxx来调用
然后可以通过trainer.xxx.xxx来调用子模块中的函数
─────────────────────────────────────────────
0.  parse_args()                                # 由 launch_train.py 注入, 含 train.yaml 路径
1.  trainer.setup.init_env(args)
       - accelerate.Accelerator(mixed_precision=bf16, log_with="mlflow", gradient_accumulation_steps=...)
       - set_seed(args.seed, device_specific=True)
       # 根据train.yaml传入的赋值给seed 无传入则默认42 需要torch.cuda.manual_seed_all(seed)
       # accelerate.utils.set_seed(seed, device_specific=True)
       # 后续还需要在允许传入generator的地方显式构造generator并且传入
       - cudnn.benchmark / deterministic / tf32 / matmul_precision
       - 在 accelerator.is_main_process 下创建 runs/<run_name>/
2.  cfg = trainer.config.load_merge(args)   # FACETConfig + TrainConfig + dump snapshot
    train_cfg: ./train.yaml 即同目录  model_cfg: ./facet/config.yaml 
3.  model = FACETWanModel(cfg.facet)            # 复用 visual/model_visual.py:225 的相同入口
       - _load_base_components / _freeze_base / _init_lora 都在 __init__ 内
       - cfg.facet.gradient_checkpointing 来自 yaml
    # 还涉及将 model 放置在 acc上 & 设置模型中仅lora参数存在梯度 其余部分全部冻结 
4.  trainer.sum.trainable_stats(model, logger)
        # 输出: q/k/v/o/ffn.0/ffn.2 各自占多少参数, 总计/可训/比率  model_visual.py中有类似可以作为参考
        # 但请注意 我不需要模型树 因为太占视觉空间了
5.  train_loader, val_loader, train_sampler, val_sampler = build_loaders(
       cfg_path="data/config.yaml",
       batch_size=cfg.train.batch_size,
       num_workers=cfg.train.num_workers,
       seed=args.seed,
       rank=accelerator.process_index,
       num_replicas=accelerator.num_processes,
   )    # 可参考 loader.py & visual/loader_visual.py
6.  optimizer = trainer.optim.build_optimizer(model, cfg.train)   # 只收 lora_down/lora_up
7.  lr_scheduler = trainer.optim.build_lr_scheduler(optimizer, cfg.train, total_steps)
8.  model, optimizer, train_loader, lr_scheduler = accelerator.prepare(
       model, optimizer, train_loader, lr_scheduler)
       # val_loader 通常不 prepare, 单卡推理验证, 避免 DDP 同步开销 具体还是参考DiffSynth的架构
# 9.  trainer.ckpt.maybe_resume(accelerator, model, optimizer, lr_scheduler, cfg.train.resume_from)
    # 暂时不考虑继续训练/resume
10. trainer.logger.setup(accelerator, cfg, args, run_dir)
       - 写 config_snapshot.yaml / env.txt / params.txt # 暂时不考虑 env.txt
       - accelerator.init_trackers("facet", config=cfg.flat(), init_kwargs={"mlflow": {...}})
11. global_step = resumed_step or 0
11.* 一些其他统计指标的初始化/零初始化 例如 start_epoch = 0
12. for epoch in range(start_epoch, cfg.train.epochs):
       train_sampler.set_epoch(epoch)
       model.train()
       for batch in train_loader:
           with accelerator.accumulate(model):
               # ---- 数据准备 (这里是 OmniControl 区别于 vanilla Wan 的地方) ----
               batch = trainer.batch.prepare(batch, model, cfg)
                   # 内部做:
                   #   a) src_latents = model.encode_src_video(batch["src_video"])      ← 每步必须做
                   #      src_mask = batch["src_mask"]
                   #   b) ref_latents = model.encode_reference_image(batch["ref_img"])  ← 每步必须做
                   #   c) tgt_latents = batch["tgt_latent"]  if cached else model.encode_src_video(tgt_video)
                   #   d) prompt_embeds = batch["t5_emb"]    if cached else model.encode_prompt(caption)
                   #   e) text/ref dropout 抽样 (cfg.facet.training.text_dropout_prob / ref_dropout_prob) 现阶段暂不考虑 只留一个分叉口在这里
                   #   f) 全部 to(device=accelerator.device, dtype=model.dtype)

               # ---- FlowMatch 噪声采样 ----
               t = trainer.loss.sample_timesteps(B, sampling=cfg.facet.training.timestep_sampling)
               # sample_timesteps需要尽可能在0~1000步中进行等概率的随机采样
               noise = torch.randn_like(tgt_latents, generator=cpu_gen)
               noisy_latents, target = trainer.loss.add_noise(
                   tgt_latents, noise, t, prediction_type=cfg.facet.training.prediction_type)
                # noise需要先to dtype / device 之后 再参与 trainer.loss.add_noise中的加噪过程 

               # ---- forward ----
               out = model(
                   noisy_latents = noisy_latents,
                   timesteps     = t,
                   prompt_embeds = prompt_embeds,
                   ref_latents   = ref_latents,
                   src_latents   = src_latents,
                   src_mask      = src_mask,
               ) # 默认 return_dict=True 返回{"pred": ...}
               loss = trainer.loss.compute_loss(out["pred"], target, cfg.facet.training.loss_type)

               accelerator.backward(loss)
               if accelerator.sync_gradients:
                   grad_norm = accelerator.clip_grad_norm_(
                       (p for p in model.parameters() if p.requires_grad),
                       cfg.train.max_grad_norm)
               optimizer.step()
               lr_scheduler.step()
               optimizer.zero_grad()

           if accelerator.sync_gradients:
               global_step += 1
               # if log_everyxxx
               trainer.logger.log_step(loss, lr_scheduler.get_last_lr()[0], grad_norm, global_step)

               if global_step % cfg.train.val_every_steps == 0:
                   metrics = trainer.valid.run(model, val_loader, val_sampler, cfg, epoch, global_step)
                   trainer.logger.log_metrics(metrics, global_step)
                   trainer.ckpt.save_topk(accelerator, model, optimizer, lr_scheduler,
                                                         metrics, epoch, global_step, run_dir, cfg.train)

13. trainer.logger.finish()

在run_name文件夹中 存在多个记录子文件 
config_snapshot.yaml 是在第2步 load_merge时就dump下去; trainable_stats 不仅在终端打印显示 也许也可以把更详细的内容记录到 params.txt中
logs文件夹在项目最开始就要创建 也就是logger打印内容需要同步到其中 或者我也可以不用 我可以使用tmux开启终端
然后使用 类似于"tee > logs/log.txt 2>&1" 将打印内容同步到 log.txt文件中  tb文件需要生成记录在 logs中
trainer.logger.log_metrics负责将metrics记录到 metrics.jsonl文件中 
然后 save_topk负责将模型checkpoint保存到 ckpts 文件夹中 

上面提到了 cfg.facet.training 我觉得这部分其实可以从 facet/config.yaml中分离出来 放置到 train.yaml中
目前实验以大量快速消融实验为主 所以选择 benchmark模式 关闭deterministic

特别注意: 模型目前已经在内部每个block上包了torch.utils.checkpoint.checkpoint 
所以只需要确保 cfg.facet.gradient_checkpointing 为True 即可启用梯度检查点
不能在accelerator调用_setup_gradient_checkpointing 否则会导致NCCL卡死


===================================================
依据于DiffSynth的配置推荐:
optimizer: AdamW;    lr: 1e-4~5e-5
weight_decay: 1e-2;  betas: (0.9, 0.999)
eps: 1e-8;  max_grad_norm: 1.0
scheduler: constant with warmup;
grad_accum_steps: 1; precision: bf16;
batch_size: 1; ema: 不启用
timestep_sampling: "logit_normal"  DiffSynth使用uniform 但是logit_normal更符合WAN的训练
seed: 依据于train.yaml 否则为 42
prediction_type: "velocity"  DiffSynth使用FlowMatch的velocity预测 这里同样设置为velocity



===================================================
关于 validation & top-K
以下以global steps为单位进行记录:
每10步log一次train loss  每500步进行1次validation log validation loss
每1000步 需要在validation的时候 抽取样本10个 进行完整的视频生成 并使用轻量metrics进行指标计算(PSNR/SSIM/LPIPS/CLIPSim)
并将生成的视频保存到 runs/output_root/samples/steps_xxx/ 中
在训练结束之后 抽取最佳checkpoint 进行大范围生成 例如生成30个样本 并且计算重指标 FVD + CLIPSIM-VAR
top-k的评选指标 暂定为lpips排序
